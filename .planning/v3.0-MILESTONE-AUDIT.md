## Non-Blocking Issues (Tech Debt)

### ISSUE-1 â€” Stale field name in GenerateCommand (code smell)
- **File:** `src/main/java/org/saltations/generate/GenerateCommand.java` lines 30, 47, 53
- **Detail:** Private field `projectName` is bound to `--destination-name` option. No behavioral impact. Picocli binds by option name, not field name; field does not appear in help output.
- **Affects:** GEN-05 (readability only)
- **Action:** Rename field to `destinationName` in a future cleanup.

### ISSUE-2 â€” Double output with --dry-run --verbose in vira generate (minor bug)
- **Files:**
  - `src/main/java/org/saltations/generate/GeneratorService.java` lines 195â€“196
  - `src/main/java/org/saltations/generate/GenerateCommand.java` lines 54â€“57
- **Detail:** `GeneratorService.generate()` directly calls `out.println("Would create " + destPath)` in the dry-run branch, AND appends to `verboseLines` when `verbose=true`. `GenerateCommand` then iterates `result.verboseLines()` a second time. - **Affects:** GEN-06 + GEN-07 combined usage
- **Action:** In the dry-run branch, append to `verboseLines` instead of printing directly; let `GenerateCommand` handle all verbose output consistently.
